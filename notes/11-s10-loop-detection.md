# 拆解笔记 11 · S10 防打转与预算闸：agent 最贵的失败不是崩溃，是打转

> 崩溃有 traceback，打转只有账单。打转不抛异常、不触发任何既有熔断（max_turns 要 40 轮才到顶），每轮烧上下文和 token、产出为零，且模型自己不自知——三个终止条件对它全部失灵：自然收口靠模型自觉（它没有）、turn 熔断太钝、中断信号没人发。deer-flow 把 LoopDetection 自标 **P0 safety**，第三季从这里开刀。解法两步：**把「重复」变成可检测信号**（tool_use 归一化 hash + 滑窗计数），**把干预做成两档**（警告教自救 → 硬停止损）。

## 一、参照系：deer-flow 的 902 行，内核只有 ~150

deer-flow 实况（子 agent 核实 2026-07-12）：`loop_detection_middleware.py` **612 行、内核 ~90**——双层检测（hash 层 + 频率层「同工具类型 30 警/50 停」）、线程级 LRU、pending 警告上限、Pydantic 配置模块；`token_budget_middleware.py` **290 行、内核 ~60**——供应商 `usage_metadata` 差分累加、input/output/total 三口径取最高占比。两件共用同一套「双档阈值 + 延迟注入」范式。

两处值得抄判断的细节：①作者把 **18-38 行的延迟注入注释自标为全文件最有教学价值的 20 行**——那是被 API 400 教育出来的经验；②注册顺序语义：**Safety 必须后注册**，才能在 after 链先跑、先剥 tool_calls，再让 LoopDetection 记账——顺序反了，滑窗会把已被安全层砍掉的调用也计进去，账就脏了。

## 三、为什么这么设计（决策清单）

1. **共享基建收成基类，而不是各写各的**。TwoTierGuard 把「双档阈值 + 延迟注入 + 剥 tool_use」焊成一件，子类只答一个问题（`_measure`：这一轮的计量值是多少）+ 两段文案。这不违反反模式 4（禁无消费者的扩展点）——**两个真实消费者当场就位**，抽象是被第二住户逼出来的，不是预留的。「TokenBudget 是同一范式的参数化」从口号变成了字面的类继承。
2. **警告延迟注入 = 排队 + 下轮 before_model**（M1）。after_model 时刻本轮 assistant 带着 tool_use、tool_result 未回填，当场 append 会夹进配对之间——API 400。这是同一条硬约束的第三个引爆点：S2 切点**避**配对 → S7 恢复**补**配对 → S10 注入**绕**配对。一句话：「检测在输出侧，干预在输入侧——中间隔着配对约束」。
3. **warn < hard 是机制推论，不是调参偏好——所以钉成构造断言**。两档重合时硬停当轮就收口，排队的警告永远等不到「下一轮 before_model」——警告根本没机会被看到，双档退化成单档惩罚。这条断言的来历值得记：它是理论课检查 2 的答案直接落进代码（教学环反哺开发第三例；第一例 S7 测试套件、第二例本刀拍板题）。
4. **硬停 = 剥 tool_use，不抛异常、不加终止分支**（M2）。剥掉后 loop 见「无工具调用」走**终止条件 1 自然收口**——复用既有终止语义，loop 零改动（C4 第十次实证）。「留/补」两个动作都有语义：**留**模型已有文本（它可能已经说了半句结论），**补**停机说明（可观测留痕，用户问「为什么停了」答得上来）。deer-flow 同款：剥 tool_calls + 改 finish_reason。
5. **归一化的不对称是信号设计的精髓**。读类**宽进**（防漏报）：只取 salient 字段、offset 按 200 行分桶——「每次换个行号刷读同一文件」归一化后仍是同一调用，逃不了检；写类**严出**（防误报）：全参入 hash——「对同一文件的多次合法小改」content 不同即不同调用，不被误杀。两个方向相反的设计放在同一个函数里，恰好证明：**信号设计的本质是误报/漏报的权衡，不是越严越好**。未识别 salient 的读工具退回全参——宁在误报侧（有警告档垫底）勿漏报。
6. **滑窗计数放实例变量，不进 State**（2026-07-13 课上拍板 A）。判据：checkpoint 保存的是「任务现场」（恢复后继续所必需），滑窗计数是**防御件的工作内存**——丢了不影响任务恢复，最坏是「刚打转到 4 次就崩溃，恢复后重新累计」，检测延迟翻倍。这是**灵敏度暂时降级，不是任务错误**，严重性差一个量级；换 State 字段则触发 S7 字段表第三次联动 + 挤占季末预算。**代价说实、知情接受。** 对抗审查（2026-07-13，0 红 6 黄）补出实例变量的另两面，已契约化进 docstring：①**一个实例 = 一个任务**——同实例跨任务复用不归零（per-instance 语义同 D5；续跑共用有意：打转前科不因续跑清零），且**不得与 subagent 共享**（排队警告会被 subagent 的 before_model 消费进隔离栈、在主对话蒸发）；②**Checkpointer 快照不含当轮待注入警告**（`_pending` 非幂等改写——checkpoint.py 的「改写幂等」注释已补例外），恢复后警告随计数一起消失，同属灵敏度降级。
7. **TokenBudget 计费 = 每个工具轮计当前 messages 总字符数**。比「只计增量」更接近真实账单形状——API 每轮**全量重提交**整个历史，历史越长每轮越贵（这也是为什么长循环的成本是超线性的）。纯文本收口轮不计费不干预——run 即将结束，干预无意义（账少记最后一轮，说实）。产品版升级只动 `_measure` 内部：换 `usage_metadata` 差分累加，缝①签名不变。

## 四、测试怎么钉住

- **全部走行为级断言，不测私有方法**：分桶归一化不直接调 `_stable_key`，而是 fixture 反证——`loop_offset_bucket`（offset 0/100/199 同桶 → 硬停）若按原始参数 hash 三次互不相同、永不触发；`loop_write_full_args`（同 path 三次不同 content → 零检出）反向钉住严出。信号设计的两个方向各有一个专属 fixture。
- **配对完整性显式断言**（`assert_pairing_intact`）：任何注入都不许把 tool_result 与它的 tool_use 拆开——把「延迟注入为什么存在」翻译成机器可查的不变量。
- **「留/补」双断言**：硬停后模型原话（「再查一次应该就有了。」）与停机说明同时在场——只断言其一，另一半语义悄悄丢了测试照样绿。
- **警告档的正反两面**：`loop_repeat_calls` 钉「警告后不改 → 硬停」，`loop_self_rescue` 钉「警告后换参数 → 计数归 1 → 自然收口」——少了后者，「教自救」就只是文案不是机制。
- **构造闸门**：`warn >= hard` 两个类各断言一次 AssertionError（理论课检查 2 的推论进了测试）。
- **fixture 复用**：预算硬停复用 S1 的 `endless_tool_calls.json`（hard=2 首轮即剥，4 条录制只消费 1 条——该 fixture 的 description 本来就写明「多余的不该被消费」）。
- **挂载次序的两个行为钉**（对抗审查黄3 落地）：`test_co_mounted_guards_strip_once_and_survive`——双件同挂时先剥者赢、后跑者走「无 tool_use 早退」跳过已剥轮（不双剥不炸，两件警告各自独立注入）；`test_interrupt_wins_when_guard_registered_before_clarification`——防御件注册在 Clarification 之前时 Interrupt 先行收口、tool_use 完好问题不丢（错误挂载序的反面行为不钉——那是配置错误，文档警告即可）。
- **未钉面说实**：滑窗淘汰（第 21 轮把第 1 轮挤出窗口、计数随之回落）没有测试——需要 20+ 轮录制，fixture 成本与教学收益不成比例；`del self._recent[:-window]` 一行逻辑靠审读（`window >= 1` 构造断言已钉住「window=0 删空切片=无限窗」的语义反转坑）。警告在 warn 与 hard 之间**每轮重复排队**（单槽覆盖，每件每轮至多一条）也未单独钉——fixture 里 warn 到 hard 只隔一轮。实例生命周期契约（跨任务复用不归零 / 不与 subagent 共享）为**文档契约未钉测试**——钉它要构造跨 run 复用的反例场景，属「错误用法的行为」，与错误挂载序同理不钉。另一条反向澄清：`test_budget_hard_stops` 的「多余录制不被消费」其实被 `turn_count == 0` 间接钉死（首轮 after_model 即剥、loop 在工具执行前收口，逻辑上只可能消费 1 条），无需补显式断言。

## 五、可迁移清单：带走的不是代码，是判断

1. **给「无声失败」造检测器**：最贵的失败往往不抛异常（打转、超支、慢性泄漏）——凡是「每轮都在花钱但没有 traceback」的失败模式，都值得一个专属信号。
2. **干预分两档，教自救优先**：警告是修复路径（模型/用户能自己改），切断是最后手段——与 S8 guard「错误信息即修复路径」同一方言。
3. **「检测在输出侧，干预在输入侧」**：凡是「看到输出才能判断、但干预必须等下一轮输入」的场景（本刀的警告、审核系统的缓拒、风控的下一单加验），中间都隔着一个「本轮不可插入」的约束——排队 + 延迟注入是通用解。
4. **信号设计 = 误报/漏报的权衡**：对逃检场景放宽匹配（分桶），对误杀场景收窄匹配（全参）——方向相反的两个设计可以且应该共存于同一个归一化函数。
5. **防御件状态 ≠ 任务现场**：持久化的划线判据是「恢复后继续任务所必需」——监控计数、缓存、账本丢了只降灵敏度不坏任务，别为它们扩持久层。

### AI PM 视角：同一批事实，另一层判断

- **双档阈值是产品参数不是工程常量**：warn/hard 的划线 = 误报率 vs 止损速度——太低误伤合法重试（用户体感：agent 动不动放弃），太高止损太慢（老板体感：账单爆炸）。两个症状方向相反，说明必须分开归因：硬停触发率高 → hard 太低；打转率高但硬停少 → hard 太高。
- **打转率 / 硬停触发率进看板**：每次硬停 = 一次未完成的任务 + 一笔白烧的钱——硬停高发场景就是下个迭代的优化清单。
- **防御动作要可观测可解释**：`[loop warning]`/`[loop stop]` 都留在 messages 里——用户问「为什么停了/为什么换路子」答得上来，这是「安全机制」变成「可运营机制」的分界线。
- **豁免的安全网 = 正交防御**：部署轮询类「合法的同参数重复」必然被单一「重复」信号误杀，产品解是 per-tool 阈值覆写（白名单做成更高阈值而非无限豁免、覆写表进新工具接入 checklist）；被豁免的工具真打转了，「花费」轴的预算闸照样兜底——**单一信号必有豁免场景，豁免的安全网在正交轴上**，这就是 S10 两件必须成对存在的产品理由。

## 六、拓展练习

1. **per-tool 阈值覆写**（部署轮询误杀场景，理论课检查 3 的现成素材）：给 `LoopDetection.__init__` 加 `thresholds_by_tool: dict[str, tuple[int, int]]`，`_measure` 后按本轮工具名取覆写档。验收：`check_deploy_status` 覆写 (20, 40) 后，10 次同参轮询零警告，而 `search` 同场景第 3 次即警告。体会「产品参数下放到工具粒度」的实现成本有多低——低到说明它被砍是因为教学聚焦，不是因为难。
2. **警告自救成功率 eval**（接 S9 的量表）：把 `loop_self_rescue` 场景做成带标签案例集（`{transcript, rescued: bool}`），用 S9 的 `evals.py` 范式量「注入警告后模型换路子的比例」——防御件的文案也该有自己的准确率账。体会「S9 装的表不只给 judge 用——一切有预期行为的组件都可以上表」。
