# 拆解笔记 12 · S11 写前版本门：状态寄生在 messages 上的确定性防御

> 这刀的立项依据是一场真实事故：deer-flow issue #3857——agent 没读文件就写、同段落追加 5 遍。机制不复杂（73 行），设计品味全在一个决策上：**读记录放哪**。答案是「哪都不放」——寄生在 `state.messages` 上。本笔记还有一段罕见的实录：**我把设计不变量写反了，是 C5 实现推演把它揪出来的**（第三节决策 2）——这个错误本身比正确答案更有教学价值。

## 一、参照系：265 行的门，与一场 5 连追加的事故

deer-flow 实况（子 agent 核实 2026-07-12）：`read_before_write_middleware.py` **265 行、内核 ~70**。立项依据 issue #3857：agent 没读文件就写，把同一段落追加了 5 遍——**防御件 roadmap 的正确来源是事故复盘，不是安全清单大全**。产品版的工程化：`additional_kwargs` 盖章（read 时把 hash 记进消息元数据、写时直接查，省掉重扫）、并发锁、async 双轨、多沙箱路径兼容。

## 三、为什么这么设计（决策清单）

1. **「读过」存内容 hash，不存 `set[path]`**。路径集合只拦得住**盲写**（没读就写）；hash 还拦得住**读旧版**（读过但磁盘被外部改了——你以为你知道的内容已经过期）。两种事故、一个门。免费赠品：**「写不刷新 mark」零实现行**——写成功后磁盘已变，messages 里旧 read 的 hash 自动失配，连续两写之间必须重读。机制的正确性自己长出了第二条规则，这是好设计的信号。
2. **读记录寄生 messages（本刀的灵魂）+ 纠错实录**。不开 State 字段（out_of_scope 明令禁止）、不用实例变量——**messages 是唯一同时具备「随 Summarization 压缩失效 + 随 checkpoint 存活」两个性质的介质，因为它就是模型记忆本身**。把门的状态和模型的记忆绑在同一介质上，两个语义自动对齐，零协调代码。**纠错实录（2026-07-13）**：三件套初稿把设计不变量写成「压缩后盲写**放行**」——这是错的，且错得隐蔽（「门随压缩自动失效」听起来像很妙的联动）。C5 实现推演时才发现：门逻辑是「找不到记录 → 拦」，压缩删记录 → **拦、逼重读**；「放行」若成立，压缩就成了绕门漏洞（模型学会等压缩后随便写）。正确语义：**门的记忆与模型的记忆同生共死——模型忘了，门跟着忘，于是逼重读**；「压缩后还放行」恰恰是实例变量方案的病（门替模型记得已被遗忘的内容）。教训：**设计不变量的表述必须先过机制推演再进文档**——「听起来很妙的联动」要跑一遍代码逻辑才算数。此错已传播到 SPEC/考点清单/课件 quiz（用户按错误答案「答对」一题），全部改正 + Artifact 重发 + 面试重考。
3. **三方对照：「状态放哪」的完整决策图谱**。S5 todos 把状态**搬出** messages（逃离压缩，计划不能丢）/ S11 读记录**藏进** messages（享受压缩，记录该随记忆过期）/ S10 滑窗放**实例变量**（防御件工作内存，连 State 都不进）。推论：寄生 State 的状态免费获得持久化（checkpoint 恢复后门的记忆完好——对照 S10 滑窗恢复归零），寄生实例的状态免费获得隔离。**放哪不是习惯，是选你要的生命周期语义。**
4. **新文件放行**：没有旧内容可覆盖，盲写无损——拦了纯误伤，防御面收窄到真实风险（与 S10 write 全参严出同思路）。注意这不是 fail-open：技术上「读不出 hash 所以没法比」是结果不是理由，理由是这里根本没有风险需要防。
5. **fail-open + 留痕（2026-07-13 课上拍板 A，记录 0022）**。判据「误拦 vs 漏过哪个更贵」：版本门防**模型犯错**（漏过最坏 = 一次可恢复的盲写，退化回没有门的世界）；误拦最坏 = 任务卡死 + **「死循环指路牌」**——门叫模型先 read，可门读不了的文件模型多半也读不了：read → 失败 → 再被拦 → 再被指路，教自救方言彻底失灵（用户的自产论证：这本身就是一种打转）。对照 S8 guard fail-closed（防越权，漏过更贵）——同一家族、两种失效取向，按防御目标选。**拍板讨论直接改了实现**：fail-open 分支加 `[version-gate bypassed: …]` 前缀留痕（静默放行的缓解，P3 可观测）——教学环反哺开发第四例，从课上合议到落进代码不到两小时。
6. **比对「模型看到的同一视图」**：比渲染后文本的 hash 而非原文，省掉「从带行号文本还原原文」的逆向工程；渲染函数从 tools.py 抽出共用，防双份漂移。**措辞据实（审查黄2）**：初稿写「render 单射故等价于比原文」——实测为假（splitlines 吞尾换行、归一化 \r\n/U+2028），render 只是**近似单射**：换行符级差异不可辨，漏检落在 fail-open 侧（危害小），足够本门语义——但数学声称要改口。

## 四、测试怎么钉住

- **6 个 fixture 各钉一个面**：自救弧（教学式 error + 终态=读后版本）/ 写不刷新 mark（磁盘停在第二版）/ 跨 run 外部改档（「录制=全局调用序」跨 run 消费）/ **设计不变量纠错版**（见下）/ fail-open 留痕 / 新文件干净放行（断言 `startswith("已写入")`——无拦截也无 bypass 前缀）。
- **设计不变量的反证链设计**（`gate_summarization_decay`）：手造历史里的 read 配对用**真实渲染**（`render_numbered(磁盘内容)`）——hash 本匹配、磁盘未变，**若无压缩本可放行**；挂上 Summarization 压掉记录后写被拦。被拦原因被反证链唯一锁定为「记录被压缩删除」，不是 hash 失配也不是没读过。断言链：摘要消息存在（压缩真发生）→ 首写被拦 → 重读后放行 → 终态=重读后版本。
- **真实的红**：首跑 `gate_summarization_decay` 弹尽——压缩后消息涨回 11 条 > max_messages=10，**Summarization 第二次触发把收口录制当摘要吃掉**（S10 同款事故的姐妹版）。修法=阈值 10→12。「弹尽 = fixture 覆盖不足」的教学样本清单又添一例。
- **对抗审查（2026-07-13，1 红 5 黄）落地**：🔴红1=**ToolOutputBudget 截断 × 版本门 = 永久误拦死循环**（审查实跑复现：截断记录 hash 永不匹配磁盘，「重读→再写」也被拦——我们自己命名的「死循环指路牌」在两个内置件组合下真实现形）→ **代码修复**：检出记录含截断标注 → fail-open 留痕放行（+钉 test_truncated_read_bypasses_gate_not_deadlock，新 fixture gate_truncated_read）；说实代价：正文恰含「已截断」字样的文件会误走放行分支（漏防侧）。黄3 路径别名误拦 → `Path.resolve()` 归一化修复（read 相对路径/write 绝对路径指同一文件不再误拦；因绝对路径 fixture 不可移植，未钉测试）。黄5 → docstring 补挂载指引「ToolErrorHandling 注册在本件之后保 bypass 留痕」。黄1/黄2/黄4 文档面已全改。
- **未钉面说实**：①**bash 侧信道完全不设防**——门只看 write 类工具的 `path` 参数，模型用 `bash` 执行 `echo x > file` 可以绕过整个门。这不是疏忽是分层：命令级防御是 S12 执行安全的地盘（审计件对 bash 命令分类），版本门只管「声明式写工具」——但两刀之间的空隙期，这个绕过面真实存在。②**同轮「先 read 后 write」会被误拦**（docstring 已说实）：loop 把同一响应的全部 tool_result 收集完才一起 append，write 的 wrap 执行时同轮 read 的结果还在局部变量里——保守语义，下轮重试即过，未钉测试（构造并行 read+write 的 fixture 收益低）。③**WRITE_HINTS 名字判定的误伤面**：名字含 "read" 的工具（如 `spread_sheet`）会被 `_last_read_digest` 误认作读工具——教学版工具集无此风险，说实即可。

## 五、可迁移清单：带走的不是代码，是判断

1. **状态放哪 = 选生命周期语义**：搬出（逃离宿主的生死）、藏进（与宿主同生共死）、实例变量（连持久化都不要）——先问「这份状态该跟谁一起死」，再决定放哪。
2. **把防御状态绑在被防御者的记忆介质上**：门与模型共享 messages，压缩/恢复两个语义自动对齐、零协调代码——「零协议耦合的语义联动」是介质选对了的免费赠品。
3. **不变量要过机制推演**：「听起来很妙的联动」在代码逻辑面前可能恰好是反的——写进文档前先手推一遍分支。
4. **失效取向按防御目标选**：防犯错 → fail-open（误拦更贵），防越权 → fail-closed（漏过更贵）；fail-open 必须配留痕（静默放行是它唯一的原罪，而留痕很便宜）。
5. **事故驱动立项**：每个防御件都该能回答「你防的那次事故长什么样」——issue #3857 之于版本门，就像账单爆炸之于预算闸。

### AI PM 视角：同一批事实，另一层判断

- **误拦率是防御件的核心产品指标**：每次误拦 = 一次被打断的合法操作 + 一轮多余往返，用户体感「agent 变笨变慢」——进看板按场景归因，高发场景（日志追加/模板化写入）就是豁免候选（per-path 白名单，豁免=更宽松不等于裸奔）。
- **guard 家族的一致性是产品体验**：三员文案同方言（教学式、给修复路径）——模型和用户学一次懂全家。
- **防御分层是产品架构决策**：版本门管声明式写、审计件管命令执行（S12）、越权门管能力（S8）——「一个门管所有」的诱惑要抵制，每道门有自己的事故形态和误拦画像。

## 六、拓展练习

1. **additional_kwargs 盖章升级**（走对照表路径）：read 时把 hash 记进该条 tool_result 消息的元数据字段，写时 O(1) 查——验证「盖章也在 message 上、压缩联动语义不变」，体会缓存层加在哪才不破坏语义。
2. **堵 bash 侧信道**（S12 预告题）：给 BashTool 的 wrap 加最简正则（检测 `>`/`>>`/`tee` 重定向到已存在文件）转交版本门判定——体会为什么产品版最终走「虚拟路径映射 + 沙箱」而不是正则打补丁（改写形态无穷尽：sed -i/python -c/dd……正则永远追不完）。
